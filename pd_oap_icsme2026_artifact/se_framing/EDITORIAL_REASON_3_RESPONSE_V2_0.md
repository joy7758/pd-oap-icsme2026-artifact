# Editorial Reason 3 Response v2.0

## Editorial criticism

"The connection to software engineering is weak and not convincingly established."

## Diagnosis

The v1.x manuscript foregrounded public-data governance context and artifact contents, but did not foreground the software engineering problem: how to specify, instantiate, validate, test, and reproduce operation-accountability artifacts.

## Why v1.x appeared weak

It read as a profile, schema, checker, and report package. It did not sufficiently argue that these are parts of a software engineering method for artifact construction and evaluation.

## v2.0 response

v2.0 reconstructs PD-OAP as a method centered on specification, validation, test fixtures, semantic checking, mutation testing, reproducibility, and failure diagnosis.

## SE concepts now foregrounded

- specification
- validation
- test fixtures
- semantic checking
- mutation testing
- reproducibility
- failure diagnosis

## Manuscript changes needed

- State the SE problem on the first page.
- Define the artifact class before discussing policy context.
- Present the method claim before implementation details.
- Use RQs to organize evaluation.
- Connect every contribution to a report or generated artifact.

## Evaluation evidence now available

v2.0 has 45 fixtures, 3 operation families, 12 semantic failure classes, a checker matrix with 45/45 matched outcomes, 72 mutations, 72/72 mutation detections and classifications, and unexpected_count 0.

## Remaining limitation

The manuscript still needs a rewritten outline and evaluation section. The current package supplies framing and evidence, not a new submission manuscript.
